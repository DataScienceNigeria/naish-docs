What is NAISH

The Nigeria AI Scaling Hub is a national AI infrastructure initiative to deploy and operate one high-performance GPU compute node, providing shared GPU access under a governed multi-tenant model.

Mission

NAISH provides shared compute access supporting a portfolio of seven priority national use cases plus broader access for AI startups and researchers. The single node is operated as a governed, multi-tenant resource rather than dedicated hardware for any one team.

The Compute Node

The proposed platform is the Supermicro SYS-821GE-TNHR — an 8U node built around eight NVIDIA H200 SXM5 GPUs on a single HGX baseboard.

HGX single-baseboard constraint

The eight GPUs sit on one indivisible HGX baseboard — they are not separable cards. This is a hard technical boundary and must be flagged wherever GPU splitting or a distributed cluster is discussed.

Software & Platform Stack

Two layers are documented — the DSN-selected open-source stack, and the optional vendor orchestration platform (AGH).

DSN open-source stack

The recommended baseline is a Kubernetes-centred open stack:

naish-platform.stack
orchestration   Kubernetes
scheduling      Volcano · Kueue           # batch / queueing
gpu-sharing     NVIDIA Device Plugin      # time-slicing, sub-GPU
notebooks       JupyterHub
inference       vLLM                      # continuous batching
tracking        MLflow                    # experiments / model registry
storage         MinIO                     # S3-compatible, on-node
identity        Keycloak
monitoring      Prometheus · Grafana · DCGM Exporter

AGH orchestration (optional, vendor)

The vendor offers the Africa GPU Hub (AGH) layer for multi-user enablement, job scheduling, model registry, usage metering, and billing. DSN's advisory position (INFRA-002) is to adopt AGH as the primary platform subject to agreement on cost and licence terms, since maintaining a fully custom multi-component stack carries significant engineering overhead. vLLM and Kubernetes remain compatible with and complementary to AGH.

Before adoption, NAISH should obtain full disclosure of AGH licensing, support obligations, and any data-residency / sovereignty implications for Nigerian proprietary data.

Serving & scheduling conventions

  • Inference workloads are time-sliced on GPU partitions (sub-GPU allocation).
  • Training / fine-tuning jobs run in scheduled windows via Volcano/Kueue, ideally off-peak.
  • Persistent data lives in MinIO on the node, with per-training-window staging (load → run → archive) rather than holding all datasets resident — the 7.6 TB scratch is finite.

Architecture Notes

Two architectural boundaries recur across the advisory record and must be preserved consistently in any documentation.

SXM5, not PCIe

The node uses the SXM5 form factor. SXM5 GPUs sit on a dedicated HGX baseboard with NVLink/NVSwitch, giving a full-mesh 900 GB/s interconnect suited to multi-GPU training and multi-GPU inference. A PCIe variant would be limited to ~64 GB/s over the host bus and lacks native NVSwitch.

The HGX single-baseboard boundary

The eight GPUs are one indivisible HGX baseboard — not separable cards. This is a hard technical boundary that shapes every capacity and hosting decision downstream.

The hybrid model is two parallel servers

Where a hybrid hosting model is discussed, it means two independent parallel servers, not a distributed cluster — a direct consequence of the HGX single-baseboard constraint. The intent is to fold external GPUs back into GBB once R1–R4 remediation is confirmed complete.

Never describe the hybrid model as a cluster. Two H200 baseboards cannot be joined into a single 16-GPU fabric.

Priority Use Cases

Seven priority use cases are mapped to the node. All seven fit simultaneously in pilot configuration; simultaneous production-scale deployment would approach node capacity, so phased rollout via the scheduler is recommended.

Agricultural Advisory Moderate GPU

Sector · AgricultureModel · N-ATLaS-LLM 8BVRAM · ~16 GB FP16 / ~9 GB INT8

Multilingual chatbot (WhatsApp, Telegram, IVR) for ~38M smallholder farmers in Hausa, Yoruba, and Igbo. FarmerChat pilot completed in Delta State under the LIFE-ND Project. Hub role: 1 GPU inference (time-sliced) + 1 GPU scheduled for fine-tuning windows.

Oral Reading Fluency (ORF) Assessment Moderate GPU

Sector · EducationModel · Whisper-large-v3 + scoring LLMVRAM · ~8–15 GB / pipeline

Reading-fluency assessment in local languages for ~86M early-grade learners. Primary workload is ASR plus a lightweight scoring/feedback LLM; typically batch / asynchronous. Benefits from CPU threads for audio pre-processing.

Liquidity Management for Agents Low GPU

Sector · Financial inclusionModel · LSTM / XGBoost / time-seriesVRAM · <1 GPU, time-sliced

Forecasting cash / e-float needs for ~2M mobile-money agents. Core models are lightweight; an optional LLM chatbot adds advisory-style inference. Primary need is high-bandwidth networking and CPU / data-pipeline capacity.

Self-CAIRE — Digital Health Coaching High criticality

Sector · HealthModel · 7–13B (up to ~33B) instructVRAM · 14–28 GB FP16 / 9–14 GB INT8

LLM-powered health coach in the mDoc "Kem" platform for maternal health and chronic disease, via WhatsApp and voice — vision to serve 150,000+ members (pilot ~300 women). Replaces the current GPT-4 dependency.

Highest safety criticality

Clinical fine-tuning, RLHF/DPO alignment, and safety evaluation (est. ~210 GPU-hours) are non-negotiable pre-deployment gates.

AI Point-of-Care Ultrasound (POCUS) High (training)

Sector · HealthEdge · Lumify / Android deviceHub role · Training / fine-tuning

Portable handheld ultrasound with AI interpretation for frontline workers. Edge inference runs on-device; the Hub trains / fine-tunes on Nigerian datasets and serves a cloud API for higher-complexity cases. 1–2 GPUs in batch training windows only — NVLink multi-GPU training is directly applicable. Primary constraint is dataset availability, not compute.

Malaria Intervention Copilot (ChatMRPT) Low–moderate GPU

Sector · HealthModel · Geospatial ML + 7–8B LLMVRAM · ~16–20 GB

Decision-support for malaria programme managers across a 218M at-risk population. Hybrid geospatial ML plus a conversational LLM layer; low user concurrency (<50). Heavier load is CPU / data-pipeline work, well served by 2 TB RAM and 112 threads.

Health Record Digitisation Moderate (throughput)

Sector · Health systemsModel · TrOCR / PaddleOCR + NLPTarget · DHIS2 / EHR formats

OCR + NLP conversion of paper records across ~31,000–33,000 facilities. Throughput-bound batch workload — an H200 processes thousands of pages/hour. 1 GPU, batch-scheduled. Dominant costs are data logistics and storage, not compute.

Capacity headline

Against total node capacity of 8 GPUs × 3 years × 8,760 hrs × 70% ≈ 147,168 effective GPU-hours, these three use cases consume roughly 57% over three years, leaving meaningful headroom (~43%) for the remaining four.

Headroom must not be overstated — it depends on the workload-governance framework being in place.

Why GPU-hour totals converge

Inference workloads run perpetually, so a time-sliced 1-GPU allocation produces a structurally static annual GPU-hour figure regardless of user growth within the target base. Additional users raise concurrency within the vLLM batch, not total GPU-hours.

Indicative cloud comparison

Reference rates used: ~$5.00/GPU-hour (specialised cloud tier; Lambda Labs $4.99, Crusoe $4.29); hyperscaler ceiling $10.00–$10.60/GPU-hour. Sub-20B inference is costed on a blended rate reflecting the fractional GPU slice actually used, not a flat full-GPU rate.

Deployment

Delivery, acceptance, and support terms from the UduTech–Kasi RFP response. UduTech leads technical advisory and orchestration; Kasi Cloud handles importation, logistics, and on-ground engineering.

Timeline

  • Hardware to Lagos — 12–20 weeks
  • Lagos to GBB — 3–7 days (3-day mobilisation upon award)
  • Installation, commissioning, acceptance — 7–14 days
  • Software + load testing — 7–14 days
  • Training, acceptance, handover — a few days
  • Total ~16–26 weeks

Acceptance test plan (RFP §7)

Hardware delivery acceptance; GPU benchmark (ResNet50 + transformer micro-benchmarks, ≥90% of vendor baseline, no thermal throttling); CPU throughput (60-min sustained); end-to-end pipeline (ingest → preprocess → train → infer within SLA); power/thermal stress (2-hr sustained + UPS switchover); network / cross-site orchestration; scheduler integration (Slurm / K8s / AGH + preemption); telemetry.

Hosting options under economic analysis

Three models are under analysis: (1) full deployment at GBB, (2) a hybrid / split model, and (3) a commercial data centre. The hybrid model is two independent parallel servers, not a distributed cluster (per the HGX constraint); the intent is to fold external GPUs back into GBB once R1–R4 remediation is confirmed.

A CAPEX/OPEX economic-comparison workbook across the three hosting models is in progress, pending GBB remediation cost lines and confirmed market rates.
